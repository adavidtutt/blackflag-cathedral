# Cathedral Schema

## Status

`frozen`

This file captures the core truth of the final system shape as stated by the user and bounded in-thread.

## First Principle

The system does not assume intelligence lives mainly in one dense blob of weights.

Instead, intelligence is distributed across:
- static trunk weights
- hot-swappable adapters
- generated deltas
- graph structure
- deterministic stores
- caches
- recurrent or working state
- retrieval
- routing
- consolidation over time

The system is less “a model” and more a society of memory-bearing machines.

## Physical Strata

### Cold Layer

Huge, slow, cheap, reality-bearing storage:
- object storage
- raw documents
- codebases
- logs
- images
- traces
- archived prompts
- archived outputs
- adapter artifacts
- tensor artifacts
- graph snapshots
- training corpora
- tokenized corpora
- replay buffers

### Warm Layer

Index-heavy and memory-heavy living library:
- SQL truth tables
- graph DB
- vector indexes
- lexical indexes
- adapter registry
- feature stores
- entity maps
- token maps
- cached evidence packets
- replay queues
- distilled summaries
- ranked memory shards

### Hot Layer

Fast RAM and VRAM only for what current thought needs:
- active execution model
- active adapters
- active expert shards
- active KV or recurrent state
- hot prefix cache
- hot evidence packet
- temporary merged deltas
- current graph neighborhood

## Memory Species

- Symbol memory
- Semantic memory
- Relational memory
- Deterministic memory
- Episodic memory
- Parameter memory
- Working memory

## Graph Hierarchy

- Graph 1: token graph
- Graph 2: span graph
- Graph 3: entity graph
- Graph 4: concept graph
- Graph 5: epistemic graph
- Graph 6: parameter graph

The parameter graph is explicitly about mind machinery, not only facts.

## Model Society

The Cathedral uses specialized model species:
- lane scouts
- lane judges
- packet builders
- route controllers
- synthesis models
- critic models
- consolidation models

This is not one emperor model with tools. It is a civilization of specialized minds.

## Query Path

The canonical staged path is:

1. preclassification
2. distributed scouting
3. packet shaping
4. execution assembly
5. synthesis
6. critique
7. answer plus memory write

## Learning Path

Learning is multi-speed:

### Fastest loop

Immediate writes:
- facts
- graph edges
- retrieval traces
- successful plans
- failures
- route outcomes
- adapter activation stats

### Fast loop

Session-level promotion:
- useful caches
- reusable packets
- temporary memory objects
- query families
- recurring graph neighborhoods

### Medium loop

Adapter evolution:
- recurring low-rank deltas
- merged adapter candidates
- expert shard specialization
- route-policy tuning
- scout-model refinement

### Slow loop

Core trunk consolidation:
- distill stable repeated behaviors
- compress common adapter families
- update durable priors

## Deterministic Truth Rule

Exact truth should live in deterministic systems whenever possible.

The synthesis layer should spend intelligence on:
- ambiguity
- tradeoffs
- composition
- explanation
- planning

It should not waste intelligence pretending to remember exact ledger facts that deterministic stores already hold.

## What This Does Not Change

- The Coding Foundry remains a bounded subsystem.
- This file does not define detailed implementation schemas by itself.
- This file does not collapse the Cathedral into a simpler agent stack.
